PR: Create greentic-redbutton cross-platform red-button webhook bridge
Repo

greentic-redbutton

PR Title

feat: add cross-platform red button listener with default Greentic webhook delivery

1. Objective

Create a new Rust CLI tool named greentic-redbutton that:

installs via cargo binstall greentic-redbutton

runs by executing:

greentic-redbutton

detects a USB red button by default using:

vendor_id = 32904 (0x8088)

product_id = 21 (0x0015)

key = enter

sends a webhook event to a default webhook URL

supports overriding all defaults via CLI flags and environment variables

works on:

Windows x86_64

Windows ARM64

macOS 15 x86_64

macOS 15 ARM64

Linux x86_64

Linux ARM64

automatically handles unplug/replug without restart

This tool is a small physical-ingress adapter for greentic-events-providers webhook.

5. Event behavior
Default semantic event

For v1, only emit a webhook on key down for the configured device/key.

Treat that as:

redbutton.click

This avoids duplicate events from press/release and keeps the integration simple.

Webhook payload

Send JSON:

{
  "source": "greentic-redbutton",
  "event_type": "redbutton.click",
  "vendor_id": 32904,
  "product_id": 21,
  "key": "enter",
  "timestamp": "2026-03-11T16:45:00Z"
}
Optional extra fields

It is fine to include:

{
  "device_name": "LinTx Keyboard",
  "os": "macos",
  "arch": "aarch64"
}

But keep the payload small and stable.

7. Internal traits and data types

Keep this minimal and testable.

Types
pub struct DeviceMatcher {
    pub vendor_id: u16,
    pub product_id: u16,
    pub key: ButtonKey,
}

pub enum ButtonKey {
    Enter,
    Other(String),
}

pub struct DeviceInfo {
    pub name: Option<String>,
    pub vendor_id: u16,
    pub product_id: u16,
    pub backend: &'static str,
}

pub enum ButtonEventKind {
    Down,
    Up,
}

pub struct ButtonEvent {
    pub kind: ButtonEventKind,
    pub timestamp: chrono::DateTime<chrono::Utc>,
}
Traits
pub trait DeviceBackend {
    fn list_devices(&self) -> anyhow::Result<Vec<DeviceInfo>>;
    fn connect(&self, matcher: &DeviceMatcher) -> anyhow::Result<Box<dyn DeviceStream>>;
}

pub trait DeviceStream: Send {
    fn next_event(&mut self) -> anyhow::Result<ButtonEvent>;
}

Keep next_event() blocking if that makes platform code easier; wrap it in a task/thread in runtime.rs.

8. Runtime design
Main loop

The runtime should behave like this:

resolve config
loop forever:
  discover target device
  if not found:
    log and retry after short interval
    continue

  connect to device stream
  while connected:
    read events
    if matching key down:
      send webhook event
    if device disappears / read fails:
      log disconnect
      break inner loop

  retry discovery
Reconnect behavior

This is important.

Requirements:

unplugging the button must not crash the process

replugging the button must not require restarting the process

runtime should automatically rediscover and reconnect

use a modest retry delay such as 1 second

log state changes clearly:

waiting for device

connected

disconnected

reconnected

Suggested log messages
waiting for red button device vendor_id=32904 product_id=21
connected to red button device name="LinTx Keyboard"
button press detected, sending webhook
device disconnected, retrying discovery
9. Platform-specific implementation expectations
Linux
Preferred implementation

Use the Linux input event subsystem.

Expected behavior

enumerate /dev/input/event*

inspect metadata to find matching VID/PID

listen for KEY_ENTER

emit Down/Up

Notes

support ARM64 and x86_64 equally

doctor should warn if permissions are insufficient

do not require root if avoidable

print helpful message if /dev/input/event* cannot be opened

Good enough for PR

If Linux is the cleanest backend initially, make it the reference-quality implementation.

macOS
Preferred implementation

Use HID/IOKit path that preserves per-device identity.

Requirements

filter by VID/PID

detect enter key from the matched HID device

support Intel and Apple Silicon

no Karabiner dependency

no CGEvent-only solution if that loses device identity

Doctor behavior

If macOS permissions/input access are relevant, doctor should clearly explain what to enable.

Windows
Preferred implementation

Use Raw Input or equivalent HID-capable approach that can distinguish which device emitted the key.

Requirements

identify device by VID/PID

detect Enter key from only that device

support x86_64 and ARM64

no global low-level keyboard hook as primary design if it cannot reliably distinguish device origin

Runtime note

A hidden message loop is acceptable if needed for Raw Input.

10. Error handling requirements

Use user-friendly errors.

Examples
Device not found
No matching red button found (vendor_id=32904, product_id=21). Plug it in and retry.
Permission issue
Could not read input device. On Linux you may need access to /dev/input/event*.
Webhook failure
Webhook POST failed: 500 Internal Server Error
Invalid configuration
Invalid key value "return". Supported values: enter

For v1, it is acceptable to support only enter and reject other keys until backend mapping is generalized.
